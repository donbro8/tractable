# Database Migrations

Tractable uses [Alembic](https://alembic.sqlalchemy.org/) to manage PostgreSQL schema migrations. All migration files live in `tractable/state/migrations/versions/`.

---

## When to Create a Migration

Create a new migration whenever you change the SQLAlchemy ORM models in `tractable/state/models.py`. This includes:

- Adding, renaming, or removing a column on any table
- Changing a column's type, nullability, or server default
- Adding or dropping a table
- Adding, modifying, or removing an index or constraint

If you change `models.py` without creating a migration, `alembic check` will fail in CI and your PR will be blocked.

---

## How to Create a Migration

1. Make your ORM model change in `tractable/state/models.py`.
2. Generate an autogenerated migration revision:

```bash
uv run alembic revision --autogenerate -m "short description of change"
```

A new file is created in `tractable/state/migrations/versions/`. The filename is prefixed with the next revision ID (e.g. `0004_short_description_of_change.py`).

---

## How to Review an Autogenerated Migration

Alembic's autogenerate is a useful starting point but **must be reviewed before committing**. Open the generated file and check:

- **`upgrade()`** — verify every `op.add_column`, `op.create_table`, `op.alter_column`, or `op.create_index` call matches your intended change exactly.
- **`downgrade()`** — verify the inverse operations are present and correct. Every schema change must be reversible.
- **Server defaults** — Alembic does not always emit `server_default` values. Add them manually when the column is `nullable=False` and must be backfilled on existing rows.
- **Rendered types** — confirm that `sa.Text()`, `sa.JSON()`, `sa.BigInteger()`, and `sa.DateTime(timezone=True)` match what the ORM model declares.
- **No-op drift** — if Alembic emits changes you did not intend (e.g. spurious index modifications), investigate and remove them.

After review, run the migration locally to confirm it applies cleanly (see the next section).

---

## How to Run Locally

Start the local service stack (PostgreSQL must be running):

```bash
docker compose -f deploy/docker-compose.yml up -d postgres
```

Apply all pending migrations to the local database:

```bash
uv run alembic upgrade head
```

To verify no unapplied model changes remain after applying migrations:

```bash
uv run alembic check
```

A zero exit code means the migration history is in sync with the ORM models. A non-zero exit code means there is still drift — either the migration is incomplete or a model change was made without a corresponding migration.

To roll back one revision during development:

```bash
uv run alembic downgrade -1
```

---

## How `alembic check` Works and What to Do When It Fails

`alembic check` compares the current SQLAlchemy ORM metadata (from `tractable/state/models.py`) against the state of the database after applying the full migration history. It exits non-zero if any difference exists.

**In CI**, `alembic check` runs as the `check-migrations` step immediately before `apply-migrations`. If it fails, the `integration-tests` job fails and the PR is blocked.

**What to do when `alembic check` fails:**

1. Run `alembic check` locally to confirm the failure.
2. Generate a new autogenerated migration:
   ```bash
   uv run alembic revision --autogenerate -m "fix schema drift"
   ```
3. Review the generated file (see above).
4. Apply it locally and re-run `alembic check` to confirm it now exits 0.
5. Commit the new migration file alongside your model changes.

Never edit an existing migration revision that has already been applied to any environment. Create a new revision instead.
